# 爬虫项目使用说明

## 1. 项目用途

当前爬虫用于按城市采集 BOSS 直聘岗位列表数据，并输出与 `job_info` 表结构对齐的主 CSV 文件。

当前脚本路径：

- `D:\graduation Project\Crawler\test_1.0\test1.py`

当前版本特点：

- 输入 `城市` 和 `目标岗位数量`
- 内部自动使用多类别种子关键词轮询抓取
- 根据已抓到的岗位标题自动扩展新的搜索关键词
- 使用跨批次去重缓存，避免重复岗位反复进入结果
- 输出主 CSV、原始 JSONL、调试文件和运行元数据

## 2. 目录结构

当前保留的核心目录如下：

```text
Crawler/
├─ test_1.0/
│  ├─ cache/
│  ├─ output/
│  └─ test1.py
└─ 使用说明.md
```

说明：

- `cache/`
  用于保存跨批次去重和扩词缓存，应该保留。
- `output/`
  用于保存每次运行的结果，可以定期清理。
- `test1.py`
  当前主爬虫脚本。

## 3. 依赖环境

建议环境：

- Python 3.11
- Chrome 已安装

需要安装的 Python 包：

```bash
pip install DrissionPage
```


## 4. 运行方式

进入脚本目录后运行：

```bash
python test1.py
```

程序会提示输入两个参数：

1. 城市
2. 目标岗位数量

例如：

```text
请输入城市（默认：贵阳）：guiyang
请输入目标岗位数量（默认：100）：300
```

说明：

- 如果终端中文输入有编码问题，可以优先使用英文别名：
  - `guiyang`
  - `zunyi`
  - `anshun`
  - `liupanshui`
  - `bijie`
  - `tongren`
  - `qiandongnan`
  - `qiannan`
  - `qianxinan`

## 5. 输出文件说明

每次运行都会生成一个目录：

```text
output/城市_时间/
```

例如：

```text
output/贵阳_2604132124/
```

目录中包含：

- `贵阳_2604132124.csv`
  主结果文件，字段与 `job_info` 对齐
- `贵阳_2604132124.jsonl`
  保留更多原始字段，便于调试和后续处理
- `meta.json`
  本次运行的元数据，包括城市、目标数量、使用过的关键词、扩展词、新增率统计等
- `debug/`
  每个关键词每一页的原始接口返回 JSON

## 6. 主 CSV 字段

当前主 CSV 字段顺序如下：

```text
job_title
company_name
city
work_address
education_text
experience_text
salary_text
salary_min
salary_max
job_description
publish_time
data_batch_no
job_hash
status
```

说明：

- 这套字段就是后端 `job_info` 表当前需要的核心字段。
- `publish_time` 目前列表页不稳定，当前版本多数情况下为空。
- `job_description` 当前主要由岗位技能关键词拼接得到，不是完整详情正文。

## 7. 去重机制

当前去重分两层：

### 7.1 本次运行内去重

每次抓到岗位后，会基于以下信息生成 `job_hash`：

- `securityId`
- `encryptJobId`
- `jobName`
- `brandName`
- `work_address`
- `salary_text`

相同 `job_hash` 的记录不会重复写入本次结果。

### 7.2 跨批次去重

缓存文件：

- `test_1.0/cache/贵阳_seen_hashes.txt`

作用：

- 保存历史已抓岗位的 `job_hash`
- 下次再跑同一城市时，如果命中已有 hash，就跳过

如果你希望重新采集同一城市并允许重复写入，可以手动删除对应城市的：

- `*_seen_hashes.txt`

## 8. 扩词机制

当前版本不是只靠固定关键词，而是“种子词 + 自动扩词”。

### 8.1 种子词

脚本里预置了多个类别的基础关键词，例如：

- 技术类
- 销售类
- 行政类
- 财务类
- 教育类
- 医疗类
- 服务类
- 制造类
- 物流类
- 设计运营类

### 8.2 自动扩词逻辑

每抓到一个新岗位，程序会读取岗位标题，然后：

1. 标题标准化
   去掉 `急聘、高薪、双休、五险一金` 等噪声词
2. 拆分标题
   例如：
   - `招商主管/招商主管助理`
   - `新媒体 运营专员`
3. 提取候选词
   例如：
   - `招商主管`
   - `招商`
   - `新媒体`
   - `运营`
4. 统计频次和得分
5. 把高频、高得分的词加入下一轮搜索

扩词缓存文件：

- `test_1.0/cache/贵阳_keyword_pool.json`

这个文件里会记录：

- 扩展关键词
- 出现次数
- 分数
- 来源岗位标题样本

## 9. 新增率停止机制

当前脚本不是无限抓，而是会计算每一页的“新增率”：

```text
新增率 = 新增唯一岗位数 / 本页岗位数
```

如果满足以下条件：

- 已抓取页面达到一定数量
- 连续多页新增率都很低

则认为当前城市本轮已经接近饱和，程序会提前停止。

这样做的目的是：

- 降低大量重复数据
- 减少无效翻页
- 控制运行时间

## 10. 如何维护

### 10.1 想扩大覆盖范围

优先改这几个常量：

- `MAX_PAGES_PER_KEYWORD`
- `MAX_ROUNDS`
- `MAX_NEW_PER_KEYWORD_PER_ROUND`
- `EXPANSION_LIMIT_PER_ROUND`

位置：

- `test_1.0/test1.py`

### 10.2 想优化扩词质量

优先改这几组配置：

- `NOISE_PATTERNS`
- `COMMON_SUFFIXES`
- `KEYWORD_CATEGORIES`

你可以把明显不是岗位名的词继续加入 `NOISE_PATTERNS`，例如：

- `朝九晚六`
- `接受无经验`
- `兼职`
- `集团`

### 10.3 想重置某个城市的历史状态

删除以下两个文件：

- `cache/<城市>_seen_hashes.txt`
- `cache/<城市>_keyword_pool.json`

删除后再运行，相当于该城市从零开始采集。

### 10.4 想保留历史数据

不要删除 `output/城市_时间/` 目录。

### 10.5 想清理项目

可以清理：

- `output/` 下的历史运行结果

不要随便清理：

- `cache/` 下的去重文件和扩词缓存

## 11. 当前已知限制

当前版本仍然是“高覆盖采样爬虫”，不是全量城市职位导出器。

限制包括：

- BOSS 本身是查询驱动站点，无法稳定获得真正“全城无关键词全量列表”
- 列表接口中的 `salary_text` 有时为空
- `publish_time` 列表页不稳定
- `job_description` 还不是完整详情页正文

如果后续要继续增强，优先级建议如下：

1. 增强扩词降噪规则
2. 增加详情页补抓
3. 增加 CSV 直接导入 MySQL 的脚本
4. 增加多数据源采集

## 12. 推荐维护顺序

后续开发建议按这个顺序继续：

1. 先稳定当前城市级广覆盖采集
2. 再写导入 MySQL 的脚本
3. 再做详情页补抓
4. 最后再接 Hadoop/Hive/Spark 清洗分析链路
